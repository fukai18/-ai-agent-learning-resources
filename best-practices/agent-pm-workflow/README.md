# Agent PM 实战：黑盒管理与 Token 效能优化 🚀

在 AI 驱动的软件开发（Vibe Coding）时代，人类的角色正迅速从“初级码农”转变为“AI 项目经理（Agent PM）”。本指南总结了在指挥 AI 执行复杂工程任务（如优化模型开发、数据流水线构建）时的核心管理心得，旨在解决 **Token 爆炸**、**API 限速**以及**管理过载**等典型痛点。

---

## 1. 黑盒管理哲学 (Black-box Management)

### ❌ 误区：过度监工 (Micromanagement)
很多开发者习惯像审阅新人代码一样，让 AI 写完一段就立刻 `read` 全量文件进行查阅。这在 Agent 协作中是极低效的：
- **Token 损耗**：频繁读取大文件会迅速堆积上下文，导致后续对话成本飙升。
- **干扰思维**：过多的代码细节会分散 PM 对业务逻辑宏观把控的注意力。

### ✅ 策略：结果导向 (Output-Oriented)
- **只信 Exit Code**：只要 AI 运行测试脚本返回 `exit 0`，就默认其逻辑正确。
- **抽样校验**：验证数据文件（如 CSV/JSON）时，严禁读取全文件。使用 `head -n 5` 或 `tail` 观察格式即可。
- **报错截断**：只有当 AI 连续 3 次无法修复 Bug 时，PM 才介入并读取受影响的代码片段。

---

## 2. 增量交付与上下文管理

### 🧩 模块化任务拆分
不要试图让 Agent 一次性交付一个完整系统。应采用“微模块”策略：
1. **数据层 (Data Gen)**：独立生成 mock 数据并校验。
2. **逻辑层 (Optimizer)**：基于 mock 数据开发核心算法，并通过单元测试。
3. **展示层 (Visualizer)**：独立开发绘图脚本。
每个模块完成后，立即进行 `git commit`，将稳定状态固化。

### ⚡️ 跨会话 (Session) 隔离
当一个任务（如开发可视化模块）已经稳定后，开启一个新的 Session 来处理下一个任务。
- **优点**：新 Session 拥有干净的上下文额度，能显著减少因上下文过载导致的“胡言乱语”或“API Rate Limit”。
- **技巧**：在新 Session 开始时，仅通过 PRD 或一份简短的架构快照（Snapshot）告知 Agent 现状。

---

## 3. 环境先行：环境即契约

AI 往往对本地环境（库版本、环境变量）缺乏感应。
- **强制依赖校验**：在让 AI 写核心逻辑前，先让它运行一个依赖检查脚本（如 `uv pip install ...`）。
- **沙箱防御**：如果发现 AI 陷入“无权访问”或“找不到路径”的循环报错，PM 应立即介入修正工作目录或权限，而不是让 AI 继续在沙箱墙内空转。

---

## 4. 总结：高效 PM 的三字经

> **看输出，不看代码；**
> **查样本，不读全量；**
> **快迭代，及时转场。**
